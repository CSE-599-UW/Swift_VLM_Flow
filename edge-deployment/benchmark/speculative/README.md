# Speculative Decoding Benchmark — Qwen2.5-VL-7B + EAGLE3 (TensorRT-Edge-LLM)

Benchmarks **EAGLE3 speculative decoding** for **Qwen2.5-VL-7B-Instruct** on the project's
**GB10** (Grace-Blackwell / DGX Spark, aarch64, sm_121) using NVIDIA
[TensorRT-Edge-LLM](https://github.com/NVIDIA/TensorRT-Edge-LLM) v0.7.1, following the
[speculative-decoding example](https://nvidia.github.io/TensorRT-Edge-LLM/latest/user_guide/examples/speculative-decoding.html).

EAGLE verifies draft tokens against the base model, so it is (near-)lossless — the headline
metric is **decode speedup**, not accuracy. We compare the same base model decoding
autoregressively (**SD-off**) vs with the EAGLE3 draft (**SD-on**).

- **Base:** `Qwen/Qwen2.5-VL-7B-Instruct`
- **Draft (EAGLE3):** `Rayzl/qwen2.5-vl-7b-eagle3-sgl`
- **Precision (this run):** **bf16** LLM + fp16 visual. fp8 is deferred to a separate x86/B200
  export (fp8 ONNX export needs ~128GB CPU RAM, infeasible on the 121GB GB10).

## Results

Full report: `results/reports/report_spec_*/report_spec.md` (generated by `report_spec.py`).

**Headline (bf16, LLaVA-Bench-in-the-Wild, 60 samples, greedy, GB10):**

| Metric | Base (SD-off) | EAGLE3 (SD-on) | Gain |
|---|---|---|---|
| Decode throughput (tok/s) | 14.3 | 27.4 | **1.91×** |
| Decode latency (ms/tok) | 69.79 | 36.47 | 1.91× faster |
| Acceptance length (tokens/step) | — | 2.53 | — |
| Prefill / TTFT (ms) | 104.7 | 108.5 | — |
| Peak VRAM (GB) | 13.71 | 13.66 | −0.04 |

EAGLE3 speculative decoding delivers a **1.91× decode speedup** on Qwen2.5-VL-7B (bf16) on the
GB10 — at essentially no extra VRAM (the 1-layer draft shares the base embedding table).

**Output parity (SD-on vs SD-off):** all 60 responses share identical openings; 18/60 exact,
mean text similarity 0.774. Divergences come from SD-on and SD-off using two *separately built*
TensorRT engines (EAGLE base vs vanilla) whose kernel/precision choices differ slightly, so a
single greedy token flip cascades — not from EAGLE being lossy. A same-engine check isn't
possible because the EAGLE base engine cannot run in vanilla mode.

## Pipeline

```
HuggingFace → ONNX export (CPU, trt-export venv) → TRT engines (GB10) → llm_inference (GB10)
   base  --eagle-base ─┐
   draft              ─┤→ engines/llm (eagle_base.engine + eagle_draft.engine + d2t)  ── SD-on
   base  (vanilla)    ─┴→ engines/llm_vanilla (llm.engine)                              ── SD-off
   visual             ──→ engines/visual (visual.engine)
```

The harness drives the C++ `llm_inference` binary as a subprocess and parses its structured
`--profileOutputFile` JSON (aggregate over all requests in one input file → one engine load).

| File | Role |
|---|---|
| `config_spec.py` | model ids, engine paths, SD knobs, sample counts |
| `io_builder.py` | build `input.json` (OpenAI-style messages, images by path) |
| `output_parser.py` | parse `profile.json` (acceptance, tok/s, prefill, VRAM) + `output.json` |
| `spec_metrics.py` | speedup + acceptance helpers (reuses `efficiency/metrics.aggregate`) |
| `run_spec_trt_edge.py` | runner: load LLaVA-Bench → run llm_inference → emit JSON |
| `run_spec_all.sh` | orchestrate SD-off then SD-on |
| `report_spec.py` | speedup/acceptance figures + markdown table |
| `env.sh` | shell env for the TRT-Edge-LLM binaries (CUDA/TRT paths) |

## How to run

```bash
conda activate sd-eagle
source edge-deployment/benchmark/speculative/env.sh    # CUDA/TRT/EDGE_LLM paths
cd edge-deployment/benchmark/speculative

# full run: SD-off baseline + SD-on EAGLE3 (LLaVA-Bench, 60 samples, 256 tokens)
bash run_spec_all.sh 60 3 256

# report (speedup + acceptance figures + table)
python report_spec.py

# unit tests
python -m pytest tests/ -q
```

Single mode:
```bash
python run_spec_trt_edge.py --num_samples 60 --spec_decode      # EAGLE3
python run_spec_trt_edge.py --num_samples 60 --no-spec_decode   # vanilla baseline
```

## Metrics

| Metric | Meaning |
|---|---|
| `decode_tokens_per_sec` / `decode_latency_ms_per_tok` | decode throughput / per-token latency (headline) |
| `acceptance_length` | avg tokens accepted per verification step (EAGLE diagnostic; >1 ⇒ speedup) |
| `ttft_ms_prefill_mean/p95` | prefill latency distribution across samples (LLM prefill stage) |
| `vision_ms` | vision-encoder time per image |
| `peak_vram_gb` | peak unified memory during the run |
| `speedup` | base `ms/tok` ÷ SD `ms/tok` (computed in the report) |

## Results JSON schema (`results/speculative/{base,spec}_<run_id>.json`)

Kevin's efficiency schema extended with SD fields: `config.spec_decode`, `config.draft_model`,
`config.spec_draft_topk/step`, `summary.acceptance_length`, `summary.decode_tokens_per_sec`,
`peak_vram_gb`, and `per_sample[].predicted_answer` (for the SD-on/off parity check).

## GB10-specific notes (deviations from the page)

The reference page assumes **export on an x86 host + build/run on a Jetson Thor**. We do the
whole pipeline on one **GB10**:

1. **TensorRT** 10.13.3.9 (aarch64/cu13) extracted locally from `.deb`s (no sudo) →
   `~/trt-10.13.3.9/usr` (see `env.sh`).
2. **C++ runtime** built natively for **sm_121** — `CMakeLists.txt` upstream targets
   `80;86;89;100a;120` (excludes GB10); changed to `set(CMAKE_CUDA_ARCHITECTURES 121)`.
3. **Export** runs in a separate `trt-export` venv with the pinned stack (`torch==2.10`,
   `transformers==5.3`, etc.) on **CPU** (bf16 needs no GPU calibration), because the GB10's
   sm_121 needs torch 2.12 which conflicts with the export pins.
4. **SD-off baseline** required a separately-built **vanilla** engine (`engines/llm_vanilla`,
   no `--specBase`) — the EAGLE base engine cannot decode in vanilla mode (its tree-verification
   input bindings aren't set), which the page's SD-only example doesn't cover.
